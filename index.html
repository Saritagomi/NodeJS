Architecture & Event Loop

Core

Node.js runs on a single-threaded event loop powered by V8 (JS engine) and libuv (async I/O + thread pool). The process never blocks waiting for I/O — instead it registers callbacks and processes results when they're ready.

Event Loop — 6 Phases

timerssetTimeout / setInterval
→
pending I/Oprev-cycle I/O errors
→
idle / prepareinternal only
→
pollfetch new I/O events
→
checksetImmediate()
→
close callbackssocket.on('close')
⚡ Between every phase — microtask drain

Node drains process.nextTick queue first, then the Promise microtask queue, before moving to the next phase. This happens after every I/O callback too.

APIQueuePriorityNotes
process.nextTick()nextTick queue1 — highestRuns before any I/O, before Promise callbacks
queueMicrotask()microtask queue2Same queue as Promise.then()
Promise.resolve()microtask queue2After nextTick, before I/O
setImmediate()check phase3After current I/O poll completes
setTimeout(fn, 0)timers phase3–4OS timer resolution (~1ms) makes order non-deterministic vs setImmediate
 
// Classic execution order puzzle
setTimeout(() => console.log('A: timeout'), 0);
setImmediate(() => console.log('B: immediate'));
Promise.resolve().then(() => console.log('C: promise'));
process.nextTick(() => console.log('D: nextTick'));
console.log('E: sync');
// ▶ E → D → C → A or B (non-deterministic) → B or A

// Inside I/O callback: setImmediate always before setTimeout
fs.readFile('x', () => {
  setTimeout(() => console.log('timeout'));
  setImmediate(() => console.log('immediate')); // always first here
});

libuv Thread Pool

Blocking I/O that cannot be done asynchronously at the OS level is offloaded to libuv's thread pool: fs.* (most ops), DNS lookup via dns.lookup(), crypto heavy ops, zlib. Default pool size = 4 threads. Saturating it causes I/O queuing.

# Increase thread pool — max 128, recommended: num CPU cores
UV_THREADPOOL_SIZE=16 node server.js
⚠ Blocking the Event Loop

Synchronous CPU work (regex on untrusted input, large JSON.parse, bcrypt) stalls all requests. Use Worker Threads for CPU work. Never call fs.readFileSync / crypto.pbkdf2Sync in request handlers.

💡 Module resolution order

CJS: exact file → .js → .json → .node → directory/index.js. node_modules walks up the directory tree. Override with NODE_PATH or --conditions flag for custom export conditions.

// Concurrency limiter — no external dependency
async function pMap(items, fn, limit = 5) {
  const results = [];
  for (let i = 0; i < items.length; i += limit) {
    const batch = items.slice(i, i + limit);
    results.push(...await Promise.all(batch.map(fn)));
  }
  return results;
}

// Async iterator / generator
async function* paginate(baseUrl) {
  let url = baseUrl;
  while (url) {
    const { data, nextPage } = await fetch(url).then(r => r.json());
    yield* data;
    url = nextPage;
  }
}
for await (const item of paginate('/api/users?page=1')) {
  process(item);
}
⚠ Common async mistakes

1. Calling an async function without await — silently ignores errors.
2. forEach with async callbacks — doesn't await them. Use for...of or Promise.all.
3. Over-using Promise.all for dependent sequential tasks.

Streams & Backpressure

API
📥
Readable

Source. Two modes: flowing (push via data event) and paused (pull via read()). Prefer async iteration.

📤
Writable

Sink. write() returns false when buffer is full. Stop writing, wait for drain event. Backpressure is your responsibility.

🔁
Duplex

Both readable and writable. Sides are independent. Example: TCP sockets, WebSocket connections.

⚙️
Transform

Duplex where output is computed from input. Implement _transform(chunk, enc, cb). Examples: zlib, cipher, CSV parser.

🔴 Backpressure — #1 streams mistake

Ignoring write() returning false accumulates data in memory without bound. Always use pipeline() which handles backpressure automatically and destroys all streams on error.

import { pipeline } from 'stream/promises';
import { createReadStream, createWriteStream } from 'fs';
import { createGzip } from 'zlib';
import { Transform } from 'stream';

// pipeline: backpressure + error cleanup — always prefer over .pipe()
await pipeline(
  createReadStream('huge.csv'),
  new Transform({
    transform(chunk, _, cb) { this.push(chunk.toString().toUpperCase()); cb(); }
  }),
  createGzip(),
  createWriteStream('out.csv.gz')
);

// Async iteration over a stream (cleanest API)
const input = createReadStream('data.ndjson', { encoding: 'utf8' });
for await (const line of input) { processLine(line); }

// Web Streams ↔ Node Streams interop (Node 18+)
import { Readable } from 'stream';
const webStream  = Readable.toWeb(nodeReadable);   // ReadableStream
const nodeStream = Readable.fromWeb(webReadable);  // Node Readable

// fetch response body IS a web ReadableStream
const nodeBody = Readable.fromWeb((await fetch(url)).body);

Buffers & Binary Data

API

Buffer is a fixed-size raw memory region outside V8's heap — a subclass of Uint8Array. Essential for binary protocols, file I/O, network packets, and crypto.

// ── Construction ──────────────────────────────────────────
Buffer.from('hello', 'utf8')          // from string
Buffer.from([0x68, 0x65, 0x6c])       // from byte array
Buffer.from(arrayBuffer, offset, len)  // from ArrayBuffer (zero-copy)
Buffer.alloc(64)                       // zero-filled — safe for user output
Buffer.allocUnsafe(64)                // uninitialized — faster, never expose raw
Buffer.concat([buf1, buf2, buf3])      // merge multiple buffers

// ── Encoding ──────────────────────────────────────────────
buf.toString('hex')     // '68656c6c6f'
buf.toString('base64')  // 'aGVsbG8='
buf.toString('utf8')    // 'hello'
// Supported: utf8, ascii, hex, base64, base64url, latin1, ucs2, binary

// ── Structured binary (custom protocol / file format) ────
const buf = Buffer.allocUnsafe(16);
buf.writeUInt32BE(0xdeadbeef, 0);    // 4 bytes big-endian at offset 0
buf.writeUInt16LE(1024, 4);          // 2 bytes little-endian at offset 4
buf.writeBigInt64LE(9007199254740991n, 6); // BigInt support
const magic = buf.readUInt32BE(0);   // 0xdeadbeef

// ── Timing-safe comparison (crypto use) ──────────────────
import { timingSafeEqual } from 'crypto';
const ok = timingSafeEqual(hmac1, hmac2); // prevents timing attacks
🔴 allocUnsafe security

Buffer.allocUnsafe() may contain sensitive data from prior memory allocations. Always write before sending. Use Buffer.alloc() for any buffer exposed to users or clients.

File System

API

Three flavors: fs (callback), fs/promises (async/await), fs.*Sync. Always use fs/promises in request paths. Sync calls block the event loop for all concurrent requests.

import { readFile, writeFile, rename, stat, mkdir, rm, opendir, watch, copyFile } from 'fs/promises';

// ── Atomic write pattern ─────────────────────────────────
await writeFile(`${path}.tmp`, JSON.stringify(data));
await rename(`${path}.tmp`, path); // atomic on same filesystem

// ── Safe existence check (avoids TOCTOU race) ────────────
try {
  const s = await stat(filePath);
  if (s.isFile()) /* exists as file */;
} catch (e) {
  if (e.code !== 'ENOENT') throw e; // only ignore "not found"
}

// ── Walk directory tree (memory-efficient) ───────────────
async function* walk(dir) {
  for await (const entry of await opendir(dir)) {
    const full = `${dir}/${entry.name}`;
    if (entry.isDirectory()) yield* walk(full);
    else yield full;
  }
}

// ── Watch (recursive stable in Node 22) ──────────────────
const watcher = watch('./src', { recursive: true });
for await (const { eventType, filename } of watcher) {
  console.log(eventType, filename);
}

// ── Recursive delete / create ────────────────────────────
await mkdir('a/b/c', { recursive: true });
await rm('dist', { recursive: true, force: true });
TaskBest approach
Read large filecreateReadStream() — never load multi-GB into memory
Atomic config writeWrite to .tmp, rename()

Recursive deleterm(path, { recursive: true, force: true })
Copy filecopyFile(src, dest, fs.constants.COPYFILE_EXCL)

⚠ dns.lookup vs dns.resolve

dns.lookup() → libuv thread pool → uses /etc/hosts + system resolver. dns.resolve4() → async, goes straight to DNS server, bypasses system resolver. In containers, always use dns.resolve*() for predictable behavior.

class DataBus extends EventEmitter {

bus.listenerCount('data');

Worker Threads

Performance

Worker Threads run V8 isolates within the same process. They share memory via SharedArrayBuffer and communicate via structured clone. Unlike fork(), startup is ~2–5ms vs ~50–100ms, but a crash in a worker can destabilize the process.

import { Worker, isMainThread, parentPort, workerData, receiveMessageOnPort, MessageChannel } from 'worker_threads';

if (isMainThread) {
  // ── Main thread ──────────────────────────────────────────
  const { port1, port2 } = new MessageChannel();
  const worker = new Worker(new URL(import.meta.url), {
    workerData: { concurrency: 4 },
    transferList: [port2]   // zero-copy port transfer
  });

  worker.postMessage({ input: bigArray }, [bigArray.buffer]); // transfer ownership
  worker.on('message', result => console.log('done', result));
  worker.on('error', err  => console.error(err));
  worker.on('exit',  code => if (code) respawn());

} else {
  // ── Worker thread ────────────────────────────────────────
  parentPort.on('message', ({ input }) => {
    const result = heavyCPUWork(input); // won't block main thread
    parentPort.postMessage(result);
  });
}

// SharedArrayBuffer + Atomics for low-latency sync
const sab  = new SharedArrayBuffer(4);
const lock = new Int32Array(sab);
// Worker: Atomics.wait(lock, 0, 0) — blocks until notified
// Main:   Atomics.notify(lock, 0, 1) — wake one worker
// Both:   Atomics.add(lock, 0, 1)    — atomic increment
💡 Use a worker pool

Creating a Worker per request is expensive (~5ms startup). Use a pool of N workers and round-robin or queue work to them. piscina is the battle-tested library for this, or implement with MessageChannel per request.

Cluster

Performance 
import cluster from 'cluster';
import os from 'os';

if (cluster.isPrimary) {
  const numCPUs = os.availableParallelism(); // Node 19+ (better than os.cpus().length)
  console.log(`Primary ${process.pid}: forking ${numCPUs} workers`);

  for (let i = 0; i < numCPUs; i++) cluster.fork();

  // Auto-restart dead workers
  cluster.on('exit', (worker, code, signal) => {
    console.error(`Worker ${worker.id} died (${signal || code}), restarting`);
    cluster.fork();
  });

  // Zero-downtime rolling restart on SIGUSR2
  process.on('SIGUSR2', () => {
    const workers = Object.values(cluster.workers);
    let i = 0;
    const restartNext = () => {
      if (i >= workers.length) return;
      workers[i++].kill('SIGTERM');
    };
    cluster.on('listening', restartNext); // restart next when new worker is ready
    restartNext();
  });

} else {
  console.log(`Worker ${process.pid} started`);
  await startHTTPServer(); // same code runs in all workers
}
ℹ Cluster vs PM2 vs Kubernetes

Cluster is great for single-host scale-up. PM2 wraps cluster with process management, logs, and restart policies. In Kubernetes, let the orchestrator handle replication and health — run single-process Node per container.

Error Handling

System 
// ── Custom error with code (Node convention) ─────────────
class AppError extends Error {
  constructor(message, code, statusCode = 500, details = {}) {
    super(message);
    this.name        = this.constructor.name;
    this.code        = code;        // e.g. 'ERR_VALIDATION'
    this.statusCode  = statusCode;
    this.details     = details;
    this.isOperational = true;     // vs programming errors
    Error.captureStackTrace(this, this.constructor);
  }
}

// ── Error.cause chain (Node 16.9+ / ES2022) ──────────────
try {
  await db.query(sql);
} catch (err) {
  throw new AppError('Query failed', 'ERR_DB', 503, { cause: err });
}

// ── Process-level safety nets ─────────────────────────────
process.on('uncaughtException', (err, origin) => {
  logger.fatal({ err, origin }, 'uncaughtException');
  process.exit(1); // MUST exit — heap state is undefined
});
process.on('unhandledRejection', (reason, promise) => {
  logger.error({ reason, promise }, 'unhandledRejection');
  // Node 15+: process will exit with code 1 automatically
});

// ── Graceful shutdown ─────────────────────────────────────
async function shutdown(signal) {
  console.log(`Received ${signal}, shutting down…`);
  // 1. Stop accepting new connections
  server.close();
  // 2. Drain in-flight requests (wait max 10s)
  const timer = setTimeout(() => process.exit(1), 10_000).unref();
  // 3. Close DB connections, flush logs
  await Promise.allSettled([db.end(), redis.quit(), logger.flush()]);
  clearTimeout(timer);
  process.exit(0);
}
['SIGTERM', 'SIGINT', 'SIGHUP'].forEach(s => process.on(s, shutdown));
🔴 Never silently swallow errors

catch(() => {}) hides real bugs. At minimum, log every caught error. In Node 15+, unhandled Promise rejections crash the process — intentional. In Node 22, --disable-warning=ExperimentalWarning is available.

Security

System
🔐
Permission Model (Node 20+)

Run with --experimental-permission to restrict FS read/write, network, child process access at OS level. Granular --allow-fs-read=/app.

☠️
Prototype Pollution

Never Object.assign(target, userInput). Use Object.create(null) for hash maps. Validate with json-schema before merging. Check __proto__, constructor, prototype keys.

⏱️
ReDoS

Catastrophic backtracking on user-controlled regex stalls the event loop. Test with safe-regex. Use RE2 (linear time) for untrusted input. Set a timeout with vm.runInContext.

🔗
Supply Chain

npm audit --audit-level=high in CI. Use --ignore-scripts for untrusted installs. Lock with package-lock.json. Consider socket.dev for deep analysis.

💉
Command Injection

Never interpolate user data into shell strings. Always use execFile(binary, argsArray). Validate and sanitize file paths — prevent path traversal with path.resolve() + prefix check.

🌊
Resource Exhaustion

Set request body size limits. Rate-limit per IP. Use timeout on all external calls. Set server.maxHeadersCount, server.headersTimeout, server.requestTimeout.

// Permission model
// node --experimental-permission --allow-fs-read=/app/data --allow-net=api.stripe.com:443 server.js

// Native .env (Node 20.6+) — no dotenv needed
// node --env-file=.env --env-file=.env.local server.js

// Validate env at startup — fail fast, fail loud
const REQUIRED = ['DATABASE_URL', 'JWT_SECRET', 'PORT', 'REDIS_URL'];
const missing  = REQUIRED.filter(k => !process.env[k]);
if (missing.length) throw new Error(`Missing env vars: ${missing.join(', ')}`);

// Path traversal prevention
import path from 'path';
const BASE     = '/app/uploads';
const resolved = path.resolve(BASE, userFilename);
if (!resolved.startsWith(BASE + path.sep)) throw new Error('Path traversal!');

// HTTP security headers (manual, or use helmet)
res.setHeader('X-Content-Type-Options', 'nosniff');
res.setHeader('X-Frame-Options', 'DENY');
res.setHeader('Content-Security-Policy', "default-src 'self'");
res.setHeader('Strict-Transport-Security', 'max-age=31536000; includeSubDomains');

V8 & Memory Internals

Performance

V8 uses generational garbage collection. Objects start in Young Generation (Scavenge GC, very fast) and promoted to Old Generation (Mark-Sweep/Mark-Compact, expensive) if they survive 2 GC cycles.

🌱
Young Generation

~1–8MB. Two semi-spaces. Scavenge is ~1ms. Short-lived objects die here. Goal: keep most objects young.

🌳
Old Generation

Default max ~1.4GB (64-bit). Mark-Sweep can take 10–100ms+. Promoted objects that survive scavenge.

📌
Large Object Space

Objects >512KB go here, never moved. Avoid allocating large objects in hot paths.

🧩
Code Space

Compiled JIT code lives here. V8 JITs hot functions. Megamorphic call sites (many object shapes) de-optimize.

// ── Tune heap size ─────────────────────────────────────────
// node --max-old-space-size=4096 server.js   (4GB old gen)
// node --max-semi-space-size=64  server.js   (64MB young gen)

// ── V8 flags useful in prod ───────────────────────────────
import v8 from 'v8';
v8.setFlagsFromString('--expose-gc'); // enable global.gc() for testing

// ── Heap stats ────────────────────────────────────────────
const stats = v8.getHeapStatistics();
// { total_heap_size, used_heap_size, heap_size_limit, malloced_memory, ... }

// ── Common memory leak patterns ───────────────────────────
// 1. Closures holding large objects after use
// 2. Event listeners not removed (call .off() or .once())
// 3. Caches without TTL or size limit (use lru-cache)
// 4. Global arrays/maps that grow forever
// 5. setInterval callbacks referencing large scope

// ── WeakRef / FinalizationRegistry (Node 14+) ────────────
const cache = new Map();
const weak  = new WeakRef(largeObject);
const registry = new FinalizationRegistry(key => cache.delete(key));
registry.register(largeObject, 'myKey');
// largeObject can now be GC'd; cache entry auto-cleaned
